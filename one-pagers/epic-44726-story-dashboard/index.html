The epic was restructured in early June: automated testing was promoted to its own feature, a Future Scope feature now holds deferred work, and a Route to Production feature gathers the deployment work — its original pipeline story (45960) was briefly Done, then reopened, and now sits at Ready to Play under the Platform team, and it has grown into a full Azure deployment breakdown covering Key Vault and secrets, a DbUp migration runner, Azure Files image storage, App Service hardening, observability, auth, and CI/CD — those deployment stories have each been decomposed into delivery tasks, around sixty in total, from ingress and secret-delivery decisions through to slot-swap promotion. Route to Production also carries a Platform-team workstream — the API repository set-up, access and permissions, App Service Terraform, Azure File Sync, an on-prem sync agent, a .NET 9 build pipeline, the deployment pipeline, and connectivity and security. Under the field-app feature, production-parity stubs mirror the RP sign-off, mapping, offline mapping, workforce, and in-app audit work alongside the login and work-order stubs, and scope stories cover feature flags to hide non-MVP surfaces, photo limits and image size, a photo gallery, job-versus-man hours, and sign-off questions. Since the last review the movement has been mixed: the RP sign-off (POC) and photo-gallery stories have moved to Awaiting QA, and on the Platform side the service-connections/identity configuration and the App Service Terraform have both closed to Done. Against that, the Azure deployment workstream has taken on blocks — Key Vault and secrets, auth and security hardening, the .NET 9 build pipeline and the on-prem image-storage story have all moved to Blocked, joining the soft-delete spike, the test-environment set-up, the on-prem API implementation story, and observability and operations, so eight stories and spikes are now Blocked, up from four. The backlog stands at 73 stories and spikes across the eight features. A broad Awaiting-QA band runs across the field-app and integration features; the test strategy and the ARB presentation are Done, the two QA execution stubs for the field app and the app-to-Woodplan route sit at Ready to Play, and the standalone Woodplan back-end test stories have been removed.

Coverage is reported here from the current working branch, edit-pole-map-improvements, exactly as it stands in the tree. This branch carries 95 JVM unit tests across 20 files — completion question and audit rules, the map feature mappers, UI-state reducers and offline-state logic, the offline map pack, PMTiles archive and tile-server stack, the on-prem API client configuration, sync-summary and session-token logic, photo limits, the financial-password verifier, and completion dictation and operational-info rules. Two instrumented tests (the Room database migration test and the sample instrumented test) sit alongside them. Honest caveat, and a downward change from the previous dashboard: the six ViewModel suites (login, operative selection, work-order list, work-order overview, completion and map) and the fourteen-flow Maestro UI suite that an earlier Testing merge had restored are not present on this branch, so no story carries a Maestro chip this cycle and the unit chips reflect the logic and infrastructure suites that are actually in the tree. There is no JDK on the build host this cycle, so the JVM suite was not re-run here — the suites in the tree await their first run on this branch. The in-app audit story does carry a CompletionAuditRules suite and a resubmission flow behind it.
